# GitSpeak — Comprehensive Frontend Architecture & UI Blueprint
**Version:** 1.0 (Final Spec)
**Paradigm:** Single-Player, Local-Native, Command-Driven

---

## 1. Aesthetic Direction & "Anti-Slop" Manifesto

GitSpeak is a premium developer tool. We reject generic "AI" aesthetics (no purple gradients, no soft bubbly cards, no default Inter/Roboto). It must feel like an extension of the terminal, elevated to a modern graphical interface.

### 1.1 Typography
*   **Display / Headers:** `Geist` or `Outfit` (Bold 600/700). Distinctive, slightly technical but beautiful.
*   **Body / UI Text:** `Geist` (Regular 400/500). Clean and hyper-legible.
*   **Monospace / Code / Hashes:** `JetBrains Mono` or `Fira Code`.
*   *Rule:* Minimum font size for inputs is `16px` to avoid mobile zoom, though desktop is the primary target.

### 1.2 Color System (Deep Dark Mode)
*   **App Background (`--bg-app`):** `#0D1117` (GitHub's deepest dark, but cooler).
*   **Card Background (`--bg-card`):** `#161B22` (Slightly elevated surface).
*   **Card Hover (`--bg-card-hover`):** `#21262D`.
*   **Borders (`--border-subtle`):** `rgba(255, 255, 255, 0.1)`. 1px crisp, clean lines. No thick borders.
*   **Primary Accent (`--accent-primary`):** `#58A6FF` (Electric Blue). Used sparingly for high-impact actions, active states, and focus rings.
*   **Text Primary (`--text-primary`):** `#E6EDF3` (Crisp off-white).
*   **Text Secondary (`--text-secondary`):** `#848D97` (Muted gray).
*   **Semantic Colors:** Red `#F85149` (Destructive/Failed CI), Green `#2EA043` (Success/Passing CI), Yellow `#D29922` (Warnings/Stashed).

### 1.3 Spatial Composition & Visual Density
*   **Layout Style:** Asymmetric and split-screen where appropriate. We actively avoid 3-column equal-width generic layouts.
*   **Spacing:** High density (`VISUAL_DENSITY = 7/10`). Think "cockpit" rather than "art gallery". Small paddings within functional components, separated by 1px dividers to maximize data density on screen.
*   **Glassmorphism:** Used *exclusively* for floating elements (Command Bar, Modals) to ensure they sit above the dense dashboard while maintaining context. `backdrop-filter: blur(12px); background: rgba(13, 17, 23, 0.8)`.

## 2. Global App Shell Structure

The Next.js 14 App Router layout uses `min-h-[100dvh]` and prevents full-page scrolling in favor of localized, independent panel scrolling.

### 2.1 The Sidebar (Left)
*   **Width:** `240px` (collapsible to `64px` icon-only mode).
*   **Content:**
    *   Workspace Switcher / User Profile (top).
    *   **Agent Status Indicator:** A pulsing dot (🟢 Connected, 🔴 Disconnected) showing the real-time WebSocket connection to the Local CLI Agent.
    *   **Registered Repositories:** List of connected repos. The active one is highlighted with an `--accent-primary` left border.
    *   **Views Navigation:** Dashboard, Visual History, Settings.

### 2.2 The Main Content Area (Center/Right)
*   Occupies the remaining width.
*   Hosts the active view (Dashboard, Graph, etc.).
*   Scrolls independently of the sidebar to keep navigation pinned.

### 2.3 The Floating Command Bar (Global Bottom)
*   **Position:** `fixed bottom-8 left-[calc(50%+120px)] -translate-x-1/2 z-50` (centered relative to the main content area).
*   **Design:** Pill-shaped, Glassmorphism base. 
*   **Behavior:** The primary way the user interacts with the app. It's an intelligent input field that accepts natural language. When a command is processed, it expands upwards to show the conversation history/results, turning into a localized chat panel.

## 3. Core Pages & UI Views

### 3.1 Onboarding & Pairing (First-Time User Experience)
*   **Step 1: Landing & Auth:** Dark, brutalist landing page. Single button: "Connect GitHub".
*   **Step 2: Agent Install Prompt:** Full-screen split view. Left side: Code snippet `npm install -g gitspeak-agent && gitspeak-agent start` with a one-click copy button. Right side: A radar-like sweeping animation waiting for the local connection.
*   **Step 3: Pairing:** User enters the 6-digit code generated by their terminal into a large, monospace, 6-box input field.
*   **Step 4: Repo Selection:** A beautiful masonry grid of their GitHub repos. Selected repos get a glowing `--accent-primary` border.
*   **Transition:** Once registered, seamless Framer Motion wipe to the Main Dashboard.

### 3.2 The Unified Repo Dashboard (The "Cockpit")
This replaces `git status`, `git log`, and GitHub.com. It is incredibly dense with information but visual hierarchy remains perfectly clear.

*   **Header:** Repo Name (`owner/repo`), Active Branch badge (`feature/auth`), Sync Status ("↑ 2 commits ahead" in accent blue).
*   **Split Layout (CSS Grid):**
    *   **Left Column (Local State - 60% width):**
        *   *Uncommitted Changes:* File tree diff showing staged/unstaged files. Hovering reveals a mini inline diff.
        *   *Recent Local Commits:* Clean list, highlighting the user's recent work natively.
        *   *Stash List:* Collapsible section showing stashed work.
    *   **Right Column (Remote State - 40% width):**
        *   *Active PRs:* Minimal cards showing PR #, title, and CI status (🟢/🔴). Clickable to open the PR review modal.
        *   *Action Runs:* Live status of GitHub Actions workflows for the current branch.
        *   *Open Issues:* High-priority tags highlighted.

### 3.3 Visual History Explorer
*   Replaces `git log --graph`.
*   **UI:** Custom SVG/Canvas rendering the git tree on the left side of the screen.
*   **Interaction:** Clicking a node in the tree opens a split-pane Monaco Editor diff view on the right side, showing exactly what changed in that commit with syntax highlighting.

### 3.4 PR & Issue Intelligence Modal
*   Based on Feature 5 from the PRD ("Summarise this PR").
*   **UI:** An off-canvas slide-over or large centered modal.
*   **Content:** When a user clicks a PR card in the Dashboard, this opens showing:
    *   **AI Summary:** A 3-bullet plain English summary of the PR diff.
    *   **CI Logs:** A small terminal window showing live CI outputs if actions are running.
    *   **Action Bar:** `[Merge PR]`, `[Close]`, `[Request Review]`.

### 3.5 Settings & Configuration View
*   **UI:** Clean, single-column layout.
*   **Sections:**
    *   *GitHub Connection:* Manage OAuth scopes, disconnect.
    *   *Agent Connection:* View machine ID, pairing code, manually disconnect agent.
    *   *Preferences:* Light/Dark mode toggle (though Dark is default/enforced mostly), default branch naming conventions.

## 4. Detailed User Flows & Motion Specs

### 4.1 The "Execute Command" Flow
1.  **Input:** User types *"Commit my changes as 'fix login bug'"* in the floating command bar.
2.  **Processing State:** The command bar border pulses subtly with `--accent-primary`. A micro-animation indicates OpenRouter API processing.
3.  **Local Execution:** The backend routes this to the Local Agent.
4.  **Result Render:** The command bar expands upwards into a `ResponseCard`.
    *   Text: "✅ Committed 3 files to `feature/auth`."
    *   Action Buttons: `[Push to Origin]`, `[Undo Commit]`.
5.  **Motion:** The expansion uses spring physics (`stiffness: 300, damping: 30` via Framer Motion) to feel snappy and responsive.

### 4.2 The "Destructive Action" Preview Flow
When a user types *"Delete the auth branch and reset hard"*...
1.  **Intercept:** The Local Agent flags `reset.hard` as destructive.
2.  **UI Shift:** The entire screen dims heavily (`backdrop-filter: blur(8px) bg-black/70`).
3.  **Modal Pop:** A brutalist, red-accented `<DestructivePreview />` modal drops from the top.
4.  **Content:** "⚠️ This will permanently discard 4 commits and 12 uncommitted file changes."
5.  **Actions:** 
    *   Primary: `Cancel` (Neutral gray).
    *   Secondary: `Yes, do it` (Bright Red. Requires typing the branch name to unlock the button).

### 4.3 Setting Up CI (YAML Generation Flow)
1.  **Input:** *"Set up CI for this project"*
2.  **Analysis:** The local agent detects it's a Next.js project.
3.  **Modal:** A `<YamlPreviewModal />` opens.
    *   Left side: Human-readable breakdown ("This workflow will run on every push, install Node 20, and run npm test").
    *   Right side: Monaco editor with the generated YAML code. User can edit it directly here.
4.  **Commit:** User clicks "Commit Workflow". Agent commits it locally and pushes.

## 5. Component Inventory & Anatomy

### 5.1 `<CommandBar />`
*   **State Machine:** `idle` -> `typing` -> `processing` -> `result_expanded`.
*   **Anatomy:** Base container is glassmorphic. The input area has no visible borders when idle. Focus state reveals a 1px `--accent-primary` ring. Right side features a subtle microphone icon (future-proofing) and a send arrow.

### 5.2 `<CommitCard />`
*   **Structure:** Flex row. Avatar (left, 24px), Message (center, truncated if long), Hash (right, monospace), Time (right, muted).
*   **Hover Effect:** Background shifts to `--bg-card-hover`. Hash becomes clickable with a "copy" tooltip.

### 5.3 `<AgentStatusIndicator />`
*   **Structure:** A 8px circular `div`. 
*   **Animation:** Uses CSS `@keyframes ping` continuously while connected. If disconnected, it turns red and stops pulsing, accompanied by a subtle "Reconnecting..." text block.

### 5.4 `<CommandHistoryLedger />`
*   **Structure:** Since the Command Bar floats, we need a place for past commands. As the user executes commands, they stack vertically behind/above the command bar in a scrollable list, similar to a terminal history but formatted as rich UI cards (`<ResponseCard />`).
*   **Interaction:** Allows scrolling back in time to see exactly what GitSpeak did 2 hours ago.

## 6. Implementation Phasing for Frontend

1.  **Phase 1: Foundation & Theme:** Setup Next.js App Router, configure Tailwind config with exact HEX codes, inject `Geist` and `JetBrains Mono` fonts, build the global `Sidebar` and `AppLayout`.
2.  **Phase 2: The Command Bar:** Build the complex floating input, the expanding animation logic with Framer Motion, and the chat history feed component.
3.  **Phase 3: The Dashboard "Cockpit":** Build the data-dense widgets (Local State, Remote State). Initially populate with mock JSON data to perfect the CSS Grid and spacing, then hook up to the local agent's WebSocket events later.
4.  **Phase 4: FTUE & Modals:** Build the pairing screen, the destructive action warnings, and the Monaco YAML viewer integration.
